## 검증 절차

### 1단계: 50건 자동 인제스트

P1a-2~6 파이프라인으로 대상 리포의 실제 커밋 50건을 자동 인제스트한다.

```
project_id: 대상 리포 경로
mode: bulk
since_date: 최근 3~6개월 범위
```

**확인 항목**:

- [ ] 50건 인제스트 완료 (실패 < 5건)
- [ ] 검증 통과율 80% 이상
- [ ] ChromaDB에 40건 이상 저장

### 2단계: 검색 시뮬레이션 5건

최근 코드 변경 5건의 diff를 선택하여 `search_review_context` 호출을 시뮬레이션한다.

```
시뮬레이션 대상: 최근 1~2주 내 실제 코드 변경 5건
각 변경에 대해:
  1. diff 텍스트를 임베딩
  2. ChromaDB에서 유사도 검색 (limit: 5)
  3. 검색 결과가 해당 리뷰에서 유용한 컨텍스트인지 수동 평가
```

### 3단계: 유용성 평가

| 평가 기준                | 유용 | 무용 |
| ------------------------ | ---- | ---- |
| 관련 과거 변경 발견      | O    | -    |
| 회귀 위험 감지           | O    | -    |
| 동일 파일/함수 이력 발견 | O    | -    |
| 관련 없는 결과만 반환    | -    | O    |
| 너무 일반적인 결과       | -    | O    |
| 검색 결과 0건            | -    | O    |

### 4단계: 판정

| 결과 | 유용성 비율         | 조치                                          |
| ---- | ------------------- | --------------------------------------------- |
| 통과 | >= 60% (3/5건 이상) | 나머지 Phase 1a 진행 (P1a-7~9)                |
| 보류 | 40~60% (2/5건)      | 프롬프트 개선 또는 임베딩 모델 변경 후 재시도 |
| 실패 | < 40% (1/5건 이하)  | 접근 방식 재검토 (RAG 대신 직접 Git 분석 등)  |
