# Time Tracker 요구사항 정의

**작성일**: 2026-03-15
**버전**: 1.0
**기반**: 사용자 기획 노트 원문

---

## 1. 개요

### 1.1 목적

사용자 기획 노트를 체계적인 기능/비기능 요구사항으로 변환하여, 구현 에이전트가 즉시 개발을 시작할 수 있는 수준으로 정리합니다.

### 1.2 범위

- **포함**: 타임 트래킹, 작업 관리, 상태 전환, 히스토리, 영속화, UI/UX, 템플릿, 통계, eCount 연동 개념
- **제외**: 구체적인 eCount API 스펙 (별도 연동 단계에서 정의)

## 4. 제약사항

### 4.1 기술적 제약

- Logseq 플러그인 환경: 브라우저 기반, `@logseq/libs` API 제한
- OPFS: Chromium 계열 브라우저에서 주요 지원
- SQLite: `sql.js` 또는 WASM 기반 SQLite 사용
- UI 언어: 한국어 only (i18n 미지원). 문자열은 상수 파일로 분리 권장
- Logseq 호환: 개발 시점 최신 안정 버전 기준. `@logseq/libs`는 `^` 범위로 관리
- 배포: Phase 1~4 GitHub Releases 수동 배포, Phase 5 GitHub Actions 자동화 검토

### 4.2 비즈니스 제약

- eCount 연동: 별도 API 스펙 확인 필요 (Phase 4-5)
- 데이터 이전: 기존 time-tracker(카운터 데모)에는 실질 데이터 없음

### 4.3 시간 제약

## 6. 사용자 스토리 (대표 예시)

### US-1: 원클릭 타임 트래킹 시작

**As a** Logseq 사용자
**I want** 현재 페이지를 기반으로 버튼 한 번으로 타임 트래킹을 시작하기
**So that** 작업 전환 시 바로 시간 기록을 시작할 수 있다

**수락 기준:**

- 현재 페이지 정보가 작업 컨텍스트로 전달됨
- "시작" 버튼 클릭 시 즉시 타이머 시작
- 진행중인 작업이 있으면 자동 중단 후 새 작업 시작

### US-2: 대기/진행/보류 작업 전환

**As a** Logseq 사용자
**I want** 대기/진행/보류 중인 작업 목록을 보고 전환하기
**So that** 여러 작업 간 스위칭이 가능하다

**수락 기준:**

- 진행중 작업 1개, 대기/보류 0개 이상 표시
- 전환 시 사유(reason) 입력 필수
- 전환 시 히스토리 기록

### US-3: 잡별/기간별 기록 조회

**As a** Logseq 사용자
**I want** 잡별·기간별 진행 시간을 조회하기
**So that** 업무 시간을 분석할 수 있다

**수락 기준:**

- 기간 선택 시 해당 기간 기록 조회
- 작업(Job)별 누적 시간 표시
- 작업+카테고리별 통계 (동일 작업 내 다중 카테고리 지원 시)
