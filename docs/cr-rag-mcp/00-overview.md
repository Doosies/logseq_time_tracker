# Code Review RAG MCP -- 프로젝트 개요

## 1. 해결하려는 문제

기존 LLM 코드 리뷰는 **현재 변경된 코드**에 대한 정적 분석(코드 컨벤션, 취약점, 가독성)은 수행할 수 있지만, 사람처럼 **과거 히스토리를 기반으로 리뷰**하는 능력은 없다.

> "이 수정이 예전에 발생했던 문제를 다시 만들지는 않을까?"

이 관점의 리뷰가 불가능한 근본 원인은 **코드 변경 히스토리를 모르기 때문**이다.

프로젝트 초기에는 히스토리의 비중이 낮지만, 프로젝트가 성숙해질수록 히스토리 기반 리뷰의 비중은 점점 높아진다.

### 추가 문제: MR 데이터만으로는 부족할 수 있다

FE 부문에서 MR 기반 코드 리뷰 후 머지하는 문화는 비교적 최근에 형성되었고, 일부 프로젝트에서만 활발하다. 따라서:

- MR이 없는 프로젝트는 **커밋 기반**으로 컨텍스트를 구축해야 한다
- 커밋 diff를 raw 상태로 넣으면 검색 품질이 떨어진다
- **LLM으로 요약한 데이터**를 벡터 DB에 넣고, 메타데이터로 원본 커밋을 참조하는 구조가 필요하다

---

## 2. 솔루션 개요

MR/커밋 데이터를 **RAG(Retrieval-Augmented Generation)** 방식으로 벡터 DB에 적재하고, 코드 리뷰 시점에 **MCP 서버**를 통해 LLM에 컨텍스트를 제공한다.

```
GitLab MR/커밋 → 수집 → LLM 요약 → 검증 → 벡터 DB 적재
                                                  ↓
                              코드 리뷰 시 ← MCP 서버 ← 유사도 검색
```

활용 범위:

- **코드 리뷰**: 과거 히스토리 기반 회귀 위험 감지
- **히스토리 파악**: 특정 파일/기능의 변경 맥락 조회
- **영향 분석**: 변경 파일의 import 관계 기반 영향 범위 파악

---

## 3. 핵심 원칙

### 3-1. 검증된 것만 인덱싱

LLM이 생성한 요약이 실제 diff와 모순되면 코드 리뷰에 해가 된다. 요약 → 검증 → 통과 시에만 벡터 DB에 적재한다. 할루시네이션이 포함된 데이터는 인덱싱하지 않는다.

### 3-2. 메타데이터로 원본 참조 보장

벡터 DB에 저장되는 것은 **요약본**이다. 항상 메타데이터(commit hash, MR ID)를 통해 원본 커밋/MR을 추적할 수 있어야 한다.

### 3-3. 점진적 확장

로컬 프로토타입에서 시작하여, 단계적으로 팀 공유 서버로 확장한다. Phase 1에서 검증된 구조를 Phase 2~3에서 스케일한다.

### 3-4. 구조화 추출 우선

diff에서 팩트(파일명, 함수명, 변경 방향)를 먼저 결정론적으로 추출하고, LLM은 "왜 변경했는가"에 대한 요약만 담당한다. 팩트와 해석을 분리함으로써 검증을 용이하게 한다.

### 3-5. 검증 전에 최적화하지 않는다

시스템의 핵심 가정("임베딩 유사도 검색이 과거 관련 변경을 정확히 찾아줄 것이다")은 아직 검증되지 않았다. **Phase 1a 초반에 파이프라인이 동작하는 시점(P1a-6 완료 후)에서 검증 게이트를 통해 핵심 가정을 먼저 검증**하고, 결과가 긍정적일 때만 나머지 단계로 진행한다. 각 Phase 사이에 ROI 검증 게이트를 두어 실효성이 확인된 범위만 확장한다.

### 3-6. 점진적 지식 축적

시스템이 제공하는 컨텍스트는 3개 축으로 구성되며, 시간이 흐를수록 풍부해진다.

| 축            | 지식 유형                                   | 시점                   | 제공 가능한 리뷰                       |
| ------------- | ------------------------------------------- | ---------------------- | -------------------------------------- |
| **코드 구조** | 인터페이스 계층, 베이스 클래스, 컨벤션 패턴 | 즉시 (인제스트 전에도) | "아키텍처 규칙상 이 코드가 적절한가"   |
| **커밋 팩트** | 변경 내용(what), 기술적 추론(why 가능 시)   | 커밋 인제스트 후       | "과거에 비슷한 변경이 있었나"          |
| **MR 맥락**   | 리뷰 디스커션, 의사결정, 합의사항           | MR 연동 후 (Phase 2+)  | "이전 리뷰에서 지적된 문제가 반복되나" |

초기에는 "이 코드는 기존 A 모듈과 구조가 유사합니다" 수준이지만, MR이 쌓이면 "이전 MR #123에서 지적된 성능 이슈가 여기서도 반복될 우려가 있습니다"와 같은 정교한 리뷰가 가능해진다.

### 3-7. 사실과 추론의 명시적 구분

LLM이 커밋을 요약할 때, diff에서 확인 가능한 **사실(what)**과 기술적 맥락에서 **추론된 내용**을 명시적으로 구분한다. 추론된 내용은 본문에 `[추론된내용]` 태그를 붙여서, 검색 결과를 소비하는 LLM이 신뢰도를 판단할 수 있게 한다. 비즈니스적 이유의 추측은 금지한다.

## 5. 기존 도구와의 차별화

| 기존 도구               | 할 수 있는 것                  | 이 시스템이 추가하는 가치                        |
| ----------------------- | ------------------------------ | ------------------------------------------------ |
| `git log` / `git blame` | 특정 파일/라인의 히스토리 조회 | **의미 기반** 유사 변경 발견 (키워드를 모를 때)  |
| GitLab MR history       | 특정 MR의 디스커션 조회        | MR **간** 패턴/반복 문제 탐지                    |
| `@codebase` (Cursor)    | 현재 코드에 대한 질의응답      | **과거 변경 히스토리** 기반 회귀 위험 감지       |
| `git log --grep`        | 키워드 기반 커밋 검색          | 키워드 없이 **의미적으로 유사한** 과거 변경 검색 |

핵심 가치: "이 변경과 관련된 과거 이력을 **자동으로 찾아서** 코드 리뷰 컨텍스트로 제공"

## 6. 로드맵 요약

### Phase 1a: MVP + 핵심 가정 검증 (~3주)

- Git CLI 수집 + 구조화 팩트 추출 (hunk 헤더 수준)
- LLM 요약 + 구조 검증 (1단계만)
- ChromaDB + 기본 검색 + 기본 후처리
- **초반 검증 게이트**: 50건 인제스트 후 검색 유용성 평가 → 실패 시 중단
- MCP 서버 (search_review_context + ingest_commits + supplement_reason)

### Phase 1b: 확장 (~3주)

- TypeScript Compiler API (AST 분석)
- 커밋 그룹핑, 코드 구조 스냅샷, FileHistoryDocument
- 추가 Tools (get_file_history, search_by_topic, get_impact_analysis)
- 후처리 강화 (맥락 조합)

### Phase 2: GitLab 통합 + 중앙 벡터 DB

- GitLab API 클라이언트 (MR + 커밋 수집)
- MR 디스커션/리뷰 코멘트 처리
- 벡터 DB 서버 분리 (Qdrant)
- 멀티 프로젝트 지원

### Phase 3: 팀 공유 서버

- HTTP/SSE MCP 서버 (팀원 공유)
- 인증/권한 관리
- CI/CD 통합 (MR 머지 시 자동 인덱싱)
- 품질 피드백 루프

### ROI 검증 게이트

각 Phase 사이에 실효성 검증 조건을 두어, 가치가 확인된 범위만 확장한다.

```
Phase 1a 내부 게이트: 50건 인제스트 후 검색 시뮬레이션 5건, 유용한 결과 60%+
Phase 1a → 1b: 실제 리뷰 2주 사용, 주 1회 이상 유용한 컨텍스트 제공
Phase 1b → 2: 단일 프로젝트 안정 운영, 팀 확장 필요성 확인
```
