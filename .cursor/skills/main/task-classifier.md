# 태스크 분류기

이 Skill은 메인 에이전트가 사용자 요청을 분석하여 적절한 태스크 유형과 워크플로우를 선택하도록 돕습니다.

## 분류 프로세스

### 1단계: 키워드 분석
사용자 요청에서 핵심 키워드를 추출합니다.

### 2단계: 맥락 파악
기존 코드베이스와의 관계를 분석합니다.

### 3단계: 유형 결정
5가지 유형 중 하나를 선택합니다.

### 4단계: 워크플로우 선택
선택한 유형에 맞는 워크플로우를 적용합니다.

---

## 태스크 유형 상세

### Feature (새 기능)

**키워드**:
- 긍정: "추가", "새로운", "구현", "만들어", "생성"
- 영어: "add", "create", "implement", "new"

**판단 기준**:
- 기존에 없던 기능인가?
- 새로운 파일이 필요한가?
- 요구사항 분석이 필요한가?

**워크플로우**:
```
기획 → 구현 → QA → 문서화
```

**예시 분류**:

```markdown
✅ Feature:
- "사용자 인증 기능 추가해줘"
- "결제 모듈 구현해줘"
- "프로필 페이지 만들어줘"
- "이메일 알림 기능 추가"

❌ Feature 아님:
- "로그인이 안돼요" → Bugfix
- "코드 정리해줘" → Refactor
```

---

### Bugfix (버그 수정)

**키워드**:
- 긍정: "버그", "오류", "안됨", "깨짐", "고장", "실패"
- 영어: "bug", "error", "broken", "not working"

**판단 기준**:
- 기존 기능이 작동하지 않는가?
- 예상과 다른 동작을 하는가?
- 에러가 발생하는가?

**워크플로우**:
```
QA (재현) → 구현 → QA (검증)
```

**예시 분류**:

```markdown
✅ Bugfix:
- "로그인이 안돼요"
- "빈 배열 처리할 때 크래시나요"
- "페이지가 깨져서 보여요"
- "API 호출 시 500 에러"

❌ Bugfix 아님:
- "로그인 기능 추가" → Feature
- "로그인 속도 개선" → Refactor
```

---

### Refactor (리팩토링)

**키워드**:
- 긍정: "개선", "정리", "리팩토링", "최적화", "중복 제거"
- 영어: "refactor", "improve", "optimize", "clean up"

**판단 기준**:
- 기능 변경 없이 구조만 개선하는가?
- 성능 최적화인가?
- 코드 품질 향상이 목적인가?

**워크플로우**:
```
기획 (영향 분석) → 구현 → QA (회귀)
```

**예시 분류**:

```markdown
✅ Refactor:
- "코드 중복 제거해줘"
- "이 함수 정리해줘"
- "성능 최적화해줘"
- "복잡한 if문 단순화"

❌ Refactor 아님:
- "새 기능 추가하면서 리팩토링" → Feature
- "버그 수정하면서 리팩토링" → Bugfix
```

---

### Docs (문서)

**키워드**:
- 긍정: "문서", "README", "주석", "설명", "API 문서"
- 영어: "documentation", "docs", "comment", "README"

**판단 기준**:
- 코드 변경이 없는가?
- 문서만 작성/수정하는가?

**워크플로우**:
```
문서화
```

**예시 분류**:

```markdown
✅ Docs:
- "README 작성해줘"
- "API 문서 추가해줘"
- "이 함수에 주석 달아줘"
- "CHANGELOG 업데이트"

❌ Docs 아님:
- "기능 추가하고 문서화" → Feature
```

---

### Hotfix (긴급)

**키워드**:
- 긍정: "긴급", "당장", "핫픽스", "지금 바로", "급함"
- 영어: "urgent", "hotfix", "asap", "emergency"
- 맥락: "프로덕션", "서버 다운", "보안"

**판단 기준**:
- 프로덕션 이슈인가?
- 매우 긴급한가?
- 최소한의 테스트로 빠르게 배포해야 하는가?

**워크플로우**:
```
구현 → QA (최소) → (나중에) 문서화
```

**예시 분류**:

```markdown
✅ Hotfix:
- "프로덕션 서버 다운, 긴급 수정"
- "보안 취약점 핫픽스"
- "결제 기능 안됨, 당장 수정 필요"

❌ Hotfix 아님:
- "긴급하게 새 기능 추가" → Feature (긴급이어도)
- "빨리 버그 수정" → Bugfix (빠르지만 hotfix는 아님)
```

## 애매한 경우 처리

### 복합 요청

```markdown
요청: "사용자 인증 추가하고 기존 코드 정리해줘"

분석:
1. "사용자 인증 추가" → Feature
2. "기존 코드 정리" → Refactor

결정: Feature (주요 목적) + Refactor (추가 작업)

워크플로우:
1. Feature 워크플로우 실행
2. 완료 후 Refactor 워크플로우 실행
```

### 불명확한 요청

```markdown
요청: "로그인 관련해서 뭔가 문제가 있어요"

분석:
- 키워드 모호
- 버그인지 개선인지 불명확

행동:
1. 사용자에게 질문
   "로그인이 작동하지 않나요? (Bugfix)
   아니면 로그인 기능 개선을 원하시나요? (Refactor)"
2. 답변에 따라 분류
```

### 우선순위 규칙

여러 유형에 해당하는 경우 우선순위:

1. **Hotfix** (최우선)
2. **Bugfix** (높음)
3. **Feature** (중간)
4. **Refactor** (낮음)
5. **Docs** (가장 낮음)
