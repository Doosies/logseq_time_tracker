# TDD Gate Workflow

TDD(Test-Driven Development)를 짧은 게이트 단위로 분할하여 빠른 피드백과 명확한 진행도 추적을 제공하는 워크플로우입니다.

## 핵심 원칙

### 1. 게이트 기반 개발
- 큰 작업을 10-20개의 작은 게이트로 분할
- 각 게이트는 30분-2시간 이내 완료 가능
- 게이트마다 명확한 검증 기준 설정

### 2. Red-Green-Refactor 사이클
각 주요 영역마다 3단계 게이트:
- **Red**: 테스트 작성 (실패하는 테스트)
- **Green**: 최소 구현 (테스트 통과)
- **Refactor**: 검증 및 개선 (리팩토링 + edge cases)

### 3. 빠른 피드백
- 각 게이트 완료 시 즉시 테스트 실행
- 문제 발생 시 해당 게이트만 재작업
- 전체 진행도를 쉽게 파악

## 게이트 구성 패턴

### 기본 구조
```
GATE 1: 기반 작업 (타입, 인터페이스 정의)
  ↓
[영역 1: 핵심 로직]
GATE 2: 테스트 작성 (Red)
GATE 3: 구현 (Green)
GATE 4: 검증 및 리팩토링 (Refactor)
  ↓
[영역 2: 통합 레이어]
GATE 5: 테스트 작성 (Red)
GATE 6: 구현 (Green)
GATE 7: 검증 및 리팩토링 (Refactor)
  ↓
GATE N: 최종 통합 및 검증
```

### 게이트 타입

| 타입 | 설명 | 검증 방법 |
|------|------|----------|
| 기반 | 타입, 상수, 설정 | TypeScript 컴파일 성공 |
| Red | 실패하는 테스트 작성 | 테스트 실행 시 실패 확인 |
| Green | 최소 구현 | 해당 게이트의 테스트 통과 |
| Refactor | 검증 및 개선 | 모든 테스트 + linter 통과 |
| 통합 | 컴포넌트 연결 | 통합 테스트 통과 |
| 검증 | 최종 확인 | 전체 테스트 + 수동 확인 |

## 계획 수립 프로세스

### 1. 작업 분석
큰 작업을 독립적인 영역으로 분해:
```
예시: "폼 검증 기능"
→ 영역 1: 검증 로직
→ 영역 2: 에러 메시지
→ 영역 3: UI 피드백
```

### 2. 게이트 생성
각 영역을 Red-Green-Refactor 게이트로 분할:

```markdown
### 영역 1: 검증 로직

**GATE 2 (Red)**: 검증 함수 테스트 작성
- 목표: validate 함수 테스트
- 테스트: 기본 동작, 에지 케이스
- 검증: 테스트 실행 시 실패 (Red)

**GATE 3 (Green)**: 검증 함수 구현
- 목표: validate 함수 구현
- 검증: GATE 2 테스트 통과

**GATE 4 (Refactor)**: 검증 함수 리팩토링
- 목표: 추가 테스트 및 리팩토링
- 검증: 모든 테스트 + linter 통과
```

### 3. 의존성 순서 정렬
게이트 간 의존성 고려하여 순서 배치:
```
타입 정의 → 핵심 로직 → 통합 레이어 → UI → 최종 검증
```

## 게이트 실행 가이드

### Red 게이트 (테스트 작성)
```typescript
describe('validate', () => {
  it('유효한 입력이면 true를 반환해야 함', () => {
    const result = validate({ name: '홍길동', email: 'test@test.com' });
    expect(result.is_valid).toBe(true);
  });

  it('빈 이름이면 에러를 반환해야 함', () => {
    const result = validate({ name: '', email: 'test@test.com' });
    expect(result.is_valid).toBe(false);
    expect(result.errors).toContain('이름은 필수입니다');
  });
});
```

완료 조건: 모든 테스트가 명확하게 실패 (구현 없음)

### Green 게이트 (구현)
최소 구현으로 테스트만 통과시킵니다. 과도한 구현 금지.

완료 조건: 해당 게이트 + 이전 게이트 테스트 모두 통과

### Refactor 게이트 (검증)
1. 추가 에지 케이스 테스트 작성
2. 코드 리팩토링 (가독성, 성능)
3. 모든 테스트 + linter 통과 확인

## Todo 관리

게이트를 Todo로 변환하여 추적:
```
- [GATE 1] 타입 정의 - 인터페이스 및 타입 선언
- [GATE 2] 핵심 로직 테스트 작성 (Red)
- [GATE 3] 핵심 로직 구현 (Green)
- [GATE 4] 핵심 로직 검증 및 리팩토링
```

## 체크리스트

### Red 게이트 완료 조건
- [ ] 테스트가 명확한 의도를 표현하는가?
- [ ] 테스트가 실패하는가? (구현 없음)
- [ ] 테스트 설명이 한글로 작성되었는가?

### Green 게이트 완료 조건
- [ ] 해당 게이트의 모든 테스트가 통과하는가?
- [ ] 이전 게이트의 테스트도 통과하는가?
- [ ] 최소 구현으로 충분한가? (과도한 구현 지양)

### Refactor 게이트 완료 조건
- [ ] 모든 단위 테스트가 통과하는가?
- [ ] linter 경고가 없는가?
- [ ] 코드 중복이 제거되었는가?
- [ ] 에지 케이스가 커버되었는가?

### 최종 검증 게이트
- [ ] 모든 단위 테스트 통과
- [ ] 모든 통합 테스트 통과
- [ ] 성능 요구사항 충족

## 베스트 프랙티스

1. **게이트는 작게**: 각 게이트 1-2시간 이내 완료
2. **독립성 유지**: 각 게이트 독립적으로 검증 가능
3. **실패는 빠르게**: Red 게이트에서 명확히 실패 확인
4. **과도한 구현 금지**: Green 게이트에서는 최소 구현만
5. **문서화**: 각 게이트의 목표와 검증 기준 명시
