# 자기 진화 멀티 에이전트 시스템 아키텍처 가이드

> 이 문서는 LLM 기반 멀티 에이전트 개발 시스템의 **설계 철학과 아키텍처 패턴**을 기술합니다.  
> 특정 프로젝트에 종속되지 않으며, 새로운 프로젝트에서 유사한 시스템을 구축할 때 참고할 수 있습니다.

---

## 1. 시스템 컨셉

### 멀티 에이전트란

하나의 만능 에이전트 대신, **역할별 전문 에이전트**를 두고 **메인 에이전트(조율자)**가 조율하는 구조입니다.

```
사용자 → 메인 에이전트 (시니어 개발자)
            ├── 기획 에이전트 → 요구사항 분석, 아키텍처 설계
            ├── 구현 에이전트 → 코드 작성, 리팩토링
            ├── QA 에이전트 → 테스트 작성, 품질 검증
            ├── 보안 에이전트 → 취약점 스캔, 입력 검증
            ├── Git 에이전트 → 커밋 메시지, PR 설명
            └── 문서 에이전트 → JSDoc, README, CHANGELOG
```

### 왜 멀티 에이전트인가

- **컨텍스트 효율**: 각 에이전트는 자기 역할에 필요한 지식만 로드
- **품질 게이트**: 에이전트 간 핸드오프 시 검증 단계 삽입 가능
- **독립적 개선**: 특정 영역의 성능만 선택적으로 개선 가능
- **병렬 실행**: 독립적인 에이전트는 동시에 작업 가능

### 자기 진화

시스템이 **자기 작업을 측정하고, 측정 결과를 바탕으로 자기 규칙을 개선**합니다.

```
작업 실행 → 메트릭 수집 → 패턴 분석 → 규칙/스킬 자동 개선 → 다음 작업에 반영
```

## 2. 3계층 지식 분리 모델

이 시스템의 **핵심 아키텍처 결정**입니다. 에이전트가 참조하는 지식을 3개 계층으로 분리합니다.

### 계층 정의

```
┌─────────────────────────────────────────────────────────┐
│  계층 1: 에이전트 정의                                      │
│  "무엇을 지켜야 하는가" (원칙)                                │
│  역할, 프로세스, 품질 게이트                                  │
│  예: "스토리지 마이그레이션 시 삼중 일치 확인"                    │
├─────────────────────────────────────────────────────────┤
│  계층 2: 도메인 스킬                                       │
│  "어떤 패턴으로 해결하는가" (기법)                              │
│  재사용 가능한 기술 패턴, 체크리스트                             │
│  예: "FOUC 방지를 위해 동기+비동기 2단계 초기화 적용"             │
├─────────────────────────────────────────────────────────┤
│  계층 3: 프로젝트 지식                                      │
│  "이 프로젝트에서는 구체적으로 어떻게 되어있는가" (구현 사실)        │
│  함수명, 초기화 순서, 테스트 패턴                               │
│  예: "initializeThemeSync()이 1단계, initializeTheme()이 2단계" │
└─────────────────────────────────────────────────────────┘
```

### 왜 3계층인가

**문제**: 에이전트 정의 파일에 프로젝트 특화 코드(함수명, 변수명 등)가 하드코딩되면:
- 코드가 변경될 때 에이전트 파일도 수정해야 함
- 에이전트 파일이 비대해져 컨텍스트 윈도우 낭비
- 다른 프로젝트에 에이전트를 재사용할 수 없음

**해결**: 각 계층의 책임을 명확히 분리

| 질문 | 계층 1 (에이전트) | 계층 2 (도메인 스킬) | 계층 3 (프로젝트 지식) |
|------|-------------------|---------------------|----------------------|
| 다른 프로젝트에서도 유효? | YES | YES | NO |
| 함수명/변수명 포함? | NO | NO | YES |
| 변경 빈도 | 거의 없음 | 거의 없음 | 코드 변경 시 |
| 컨텍스트 로드 | 항상 | 관련 작업 시 | 해당 패키지 작업 시 |

### 판단 규칙

내용을 어느 계층에 넣을지 결정하는 질문:

1. **"함수명/변수명을 제거해도 의미가 통하는가?"**
   - YES → 계층 1 또는 2
   - NO → 계층 3 (프로젝트 지식)

2. **"이 내용은 원칙인가, 기법인가?"**
   - 원칙 (1~2줄로 요약 가능) → 계층 1 (에이전트 정의)
   - 기법 (체크리스트, 코드 패턴) → 계층 2 (도메인 스킬)

### 실전 규칙

- 에이전트 정의에는 **원칙 1~2줄 + 스킬 참조 링크**만 남긴다
- 도메인 스킬의 코드 예시에서는 **제네릭 이름** 사용 (`initializeX`, `state_var` 등)
- 프로젝트 지식은 **패키지 단위**로 조직한다

## 3. 에이전트 일반화 원칙

### 에이전트 정의 파일의 구조

```markdown
---
role: 역할명
responsibilities: [책임 목록]
skills: [참조할 스킬 파일]
name: 에이전트 식별자
---

# 에이전트 제목
> 역할, 목표, 원칙 (1줄씩)

## 핵심 원칙 (범용)
## 역할 (입력/출력/책임)
## 품질 기준 (범용 체크리스트)
## 참조 스킬 (구체 기법은 스킬에 위임)
## 주의사항 (범용 원칙만)
## 작업 완료 조건
```

### 에이전트에 넣어야 할 것

- 역할과 책임의 경계
- 범용 품질 기준 (예: "Linter 오류 0개", "테스트 커버리지 80%+")
- 다른 에이전트와의 협업 프로토콜
- 범용 원칙 (예: "가정 금지, 코드 직접 읽기")

### 에이전트에 넣지 말아야 할 것

- 특정 라이브러리의 API 사용법 → 도메인 스킬
- 특정 함수의 호출 순서 → 프로젝트 지식
- 코드 예시 (3줄 이상) → 도메인 스킬 또는 프로젝트 지식
- 프로젝트 특화 체크리스트 → 프로젝트 지식

## 4. 스킬 설계 기준

### 파일 구조

```
skills/
├── {domain}/                # 도메인별 폴더
│   ├── SKILL.md             # 진입점: 목적, 사용 시점, 스킬 목록
│   └── references/          # 세부 레퍼런스
│       ├── pattern-a.md
│       └── pattern-b.md
│
├── project-conventions/     # 프로젝트 코딩 스타일 (HOW)
│   └── SKILL.md
│
└── project-knowledge/       # 프로젝트 구현 지식 (WHAT)
    ├── SKILL.md
    └── references/
        ├── package-a.md     # 패키지별 구현 지식
        └── package-b.md
```

### SKILL.md 작성 규칙

```markdown
---
name: skill-name
description: 1줄 설명
---

# 스킬 제목

## 목적 (왜 필요한가)
## 사용 시점 (언제 로드하는가)
## 참조 구조 (references/ 파일 목록)
```

### 도메인 스킬 vs 프로젝트 지식

| | 도메인 스킬 | 프로젝트 지식 |
|---|---|---|
| 목적 | 기술 패턴 전달 | 구현 사실 전달 |
| 재사용성 | 다른 프로젝트에서도 사용 가능 | 이 프로젝트에서만 유효 |
| 코드 예시 | 제네릭 이름 (`initializeX`, `resetForTests`) | 실제 이름 (`initializeTheme`, `resetThemeForTests`) |
| 유지보수 | 패턴이 변할 때만 | 코드가 변할 때마다 |
| 조직 | 에이전트 역할별 | 패키지/모듈별 |

### `project-conventions` vs `project-knowledge`

둘 다 프로젝트에 특화되어 있지만 관점이 다릅니다:

| | project-conventions | project-knowledge |
|---|---|---|
| 관점 | HOW (어떻게 코드를 작성하는가) | WHAT (무엇이 구현되어 있는가) |
| 내용 | 네이밍 규칙, 파일 패턴, import 순서 | 스토어 초기화 순서, 함수 역할 |
| 변경 빈도 | 거의 변하지 않음 | 기능 변경 시 업데이트 |

## 5. 워크플로우 패턴

### 태스크 분류

사용자 요청을 아래 유형으로 분류하고, 유형별로 에이전트 조합 패턴을 적용합니다.

| 유형 | 에이전트 조합 | 특징 |
|------|-------------|------|
| Feature | 기획 → 보안(설계) → 구현 → 보안(코드) → QA+문서(병렬) → Git | 가장 완전한 워크플로우 |
| Bugfix | QA(재현) → 구현 → 보안 → QA(검증) → Git | 기획 생략, 재현이 핵심 |
| Refactor | 기획(영향) → 구현 → QA(회귀) → Git | 회귀 테스트가 핵심 |
| Docs | 문서화 → 승인 | 가장 단순 |
| Hotfix | 구현 → QA(최소) → 긴급 승인 | 속도 우선, 사후 문서화 |
| Chore | 구현 → type-check+test → 승인 | 설정/인프라 변경 |

### 품질 게이트

에이전트 간 핸드오프 시 메인 에이전트가 검증합니다:

- **구현 후**: Linter 0개, type-check 통과, 위험 패턴 grep
- **QA 후**: 모든 테스트 통과, 커버리지 80%+
- **Git 전**: Conventional Commits 형식, 한글 subject

### 위임 원칙

메인 에이전트는 **조율자**입니다. 코드를 직접 작성하지 않고 서브에이전트에게 위임합니다.

직접 허용: 1~2줄 오타/import 수정만.
그 외 모든 작업: 해당 전문 에이전트에 위임.

## 6. 자기 진화 메커니즘

### 메트릭 수집

각 작업 사이클에서 데이터를 수집합니다:

- **시간**: 시작/완료 시각, 에이전트별 소요 시간
- **품질**: Linter 오류, 테스트 통과율, 커버리지
- **효율**: 재시도 횟수, 에러율
- **결정**: 주요 의사결정 사항과 근거
- **이슈**: 발생한 문제와 해결 방법

### 분석 트리거

- 정기: N개 사이클마다 (예: 20개)
- 임계값: 에러율 30%+, 평균 시간 50%+ 증가, 재시도 3회+
- 수동: 사용자 요청 시

### 개선 프로세스

```
1. 패턴 감지 (동일 실수 반복, 비효율 워크플로우)
2. 원인 분석 (어느 에이전트, 어느 단계)
3. 개선안 작성 (에이전트 규칙 수정 또는 스킬 추가)
4. A/B 테스트 (10+10 사이클 비교)
5. 채택 또는 롤백
6. 개선 이력 기록
```

### 메타 에이전트

- **시스템 개선 에이전트**: Rule/Skill 최적화, 워크플로우 개선
- **MCP 개발 에이전트**: 반복 작업 감지 시 자동화 도구 개발

## 7. 새 프로젝트에 적용하기

### Phase 1: 기본 구축

1. **`AGENTS.md` 작성**: 공통 코딩 컨벤션, 에이전트 역할 요약
2. **에이전트 정의 파일 작성**: 최소 3개 (구현, QA, 문서화)
3. **메인 에이전트 Rule 작성**: 태스크 분류, 워크플로우, 품질 게이트
4. **공용 스킬 작성**: project-conventions, error-handling

### Phase 2: 스킬 확장

5. **도메인 스킬 추가**: 프로젝트에서 사용하는 기술별 패턴 문서화
6. **project-knowledge 스킬 생성**: 프로젝트 구현 사실 기록 시작
7. **워크플로우 정의**: 태스크 유형별 에이전트 조합 패턴

### Phase 3: 자기 진화

8. **메트릭 수집 구조 구축**: cycle JSON 스키마, 보고서 템플릿
9. **시스템 개선 에이전트 추가**: 메트릭 분석, Rule/Skill 자동 개선
10. **보안 에이전트 추가**: 설계/구현/배포 단계별 검증

### 구축 시 주의사항

- **점진적으로 시작**: 처음부터 모든 에이전트를 만들지 않는다. 핵심 3개(구현, QA, 문서)부터.
- **실전 검증 우선**: 에이전트를 만든 후 반드시 실제 태스크로 검증하고 피드백을 반영한다.
- **규칙은 적게, 원칙은 명확하게**: 에이전트 정의가 500줄을 넘으면 스킬 분리를 검토한다.
- **프로젝트 지식은 코드와 함께 살아간다**: 코드를 변경할 때 project-knowledge도 업데이트하는 습관을 만든다.
