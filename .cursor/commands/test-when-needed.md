# 테스트 필요성 판단 및 선택적 테스트 작성 워크플로우

코드 변경 사항을 분석한 뒤 아래 6단계를 수행한다. 테스트가 필요하다고 판단되면 QA 에이전트를 호출하여 최소 유효 테스트를 작성하고, 미작성 시에는 반드시 근거를 기록하여 보고한다.

코드 변경 작업(Feature/Bugfix/Refactor)에서 **테스트가 필요하다고 판단될 때만** 테스트를 작성하는 워크플로우입니다.

## 목적

- 변경 영향에 따라 테스트 필요성을 객관적으로 판단
- 불필요한 경우: 테스트 미작성 + **근거 기록** (품질 저하 방지)
- 필요한 경우: 최소 유효 테스트 작성 (핵심 경로·회귀 위험 중심)

## 원칙

- **강제 아님**: 모든 변경에 테스트를 요구하지 않음
- **미작성 시 근거 필수**: 테스트를 작성하지 않기로 한 경우, 판단 근거를 반드시 기록

## 실행 단계

### 1단계: 변경 영향 분석

- 변경된 파일 및 함수 식별
- 기존 테스트 존재 여부 확인
- 변경 유형 분류 (로직 추가/수정, 리팩토링, 버그 수정, 설정 변경 등)

### 2단계: 테스트 필요성 점수표

| 항목 | 낮음 (0~1) | 중간 (2) | 높음 (3) | 점수 |
|------|-----------|---------|---------|------|
| 회귀 위험 | 변경 없음/설정만 | 로직 일부 수정 | 핵심 로직 변경 |  |
| 복잡도 | 10줄 이하 단순 코드 | 중간 복잡도 | 분기·알고리즘 복잡 |  |
| 외부 의존성 | 없음 | 일부 mock 가능 | API·DB 등 의존 다수 |  |
| UI 상호작용 | 없음 | 간단한 이벤트 | 폼·상태·비동기 복잡 |  |

**판단 기준**: 합계 6점 이상 → 테스트 작성 권장. 5점 이하 → 미작성 가능 (근거 기록)

### 3단계: 작성 여부 결정

- **작성**: 점수표 결과 + 도메인 중요도 종합 판단
- **미작성**: 점수표 결과 + "미작성 근거" 명시 (예: "단순 prop 전달 변경, 회귀 위험 거의 없음")

### 4단계: 작성 시 테스트 범위·위치·명명

- **범위**: 핵심 성공 경로 1개 + 실패/엣지 1개 이상 (최소 세트)
- **위치**: `__tests__/*.test.ts` 또는 해당 모듈 옆 `.test.ts`
- **명명**: `describe('함수/컴포넌트명')`, `it('한글 설명')` (프로젝트 컨벤션 준수)

### 5단계: 실행 및 검증

- `pnpm test` 실행
- ReadLints로 테스트 코드 검증
- 결과 기록

### 6단계: 결과 보고

아래 템플릿으로 메인 에이전트에게 보고합니다.

---

## 결과 보고 템플릿

```markdown
# 테스트 필요성 판단 결과

## 변경 영향 분석
- 변경 파일: [목록]
- 변경 유형: [Feature/Bugfix/Refactor/설정]
- 기존 테스트: [있음/없음]

## 테스트 필요성 점수표
| 항목 | 점수 | 비고 |
|------|------|------|
| 회귀 위험 | 0~3 |  |
| 복잡도 | 0~3 |  |
| 외부 의존성 | 0~3 |  |
| UI 상호작용 | 0~3 |  |
| **합계** |  |  |

## 결정
- [ ] **테스트 작성함** → 파일 위치, 테스트 개수, 실행 결과
- [ ] **테스트 미작성** → 근거 (필수!)

## 결정사항 (Decisions)
| 결정 | 근거 | 검토한 대안 |
|------|------|-------------|
|  |  |  |

## 발견된 이슈 (Issues)
| 이슈 | 해결 방법 | 영향도 |
|------|-----------|--------|
| none 또는 구체적 이슈 |  |  |
```

## 사용 방법

에이전트 입력창에 다음을 입력하여 실행:

```
/test-when-needed
```

또는 "테스트 필요성 판단해줘", "이 변경에 테스트 필요한지 평가해줘" 등으로 요청합니다.
