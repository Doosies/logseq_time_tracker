## 2. 이번 사이클에서 드러난 패턴

### 2.1 스토리지·async 초기화

- **단일 진실 공급원 이동**: 동기 `localStorage` 기반 스토어 → 비동기 `chrome.storage.sync`는 초기화 완료 시점이 늦어짐.
- **FOUC/깜빡임**: 첫 페인트 전 테마/선호를 알 수 없을 수 있어, **동기 가능한 최소 정보**와 **비동기 완료 후 정식 반영**의 2단계(`initializeThemeSync` + `initializeTheme`)가 자연스러운 구조.
- **onMount와 async**: 컴포넌트 `onMount`에서 스토어가 Promise를 반환하면 **기존 동기 가정 코드가 깨짐** — `await` 또는 “준비 완료” 가드가 필요함.

### 2.2 마이그레이션 분기와 상태 일관성 (핵심 교훈)

- 마이그레이션 분기에서 **영속 저장소만** 옮기고 **메모리상 파생 상태**(예: `current_theme`)를 갱신하지 않으면, UI와 저장소가 어긋남.
- 이번 사이클에서는 **QA 단계에서 발견 후 수정** — 명세에 “마이그레이션 시 UI와 동기화할 상태 목록”이 없었을 때 전형적으로 발생하는 클래스의 결함.

### 2.3 첫 설치·테스트 재현

- `setup_state` 등 **플래그 기반 흐름**은 테스트에서 `setup_completed`를 `undefined`로 두는 등 **초기값 시나리오**를 명시적으로 재현해야 함.
- **모듈 전역 상태**가 남는 경우 `resetTabStateForTests()` 같은 **테스트 전용 리셋 훅**으로 격리 — 회귀·순서 의존 완화.

### 2.4 Svelte 5·도구 경고

- `bind:this`에 쓰는 ref는 `$state<HTMLInputElement | undefined>(undefined)` 등으로 두어 **경고/타입**을 동시에 정리 — Runes와 DOM ref 패턴 문서화 가치 있음.

### 2.5 보안·제품

- **파일 가져오기 크기 제한 미적용**은 Critical/High는 아니나 **권고**로 남음 — Planner/Security에 “수용 기준 또는 추적 이슈”로 남기는 것이 좋음.

## 3. 재사용 가능한 패턴: “async 스토어 마이그레이션” 체크리스트

아래는 동일 계열 작업(확장 프로그램 설정 동기화, IndexedDB/스토리지 전환 등)에 **그대로 복사해 쓸 수 있는 점검표**입니다.

### 3.1 설계·구현 (Developer / Planner)

1. **초기화 API 분리**: 동기 가능한 최소 로드 vs 비동기 전체 동기화 함수명·호출 순서를 설계 문서에 명시.
2. **마이그레이션 시 상태 목록**: 옮기는 키뿐 아니라 **화면/스토어에 반영되는 모든 파생 상태**를 표로 적고, 분기마다 갱신 여부를 체크.
3. **onMount / 라이프사이클**: async 스토어를 쓰는 컴포넌트에서 `await`·로딩 상태·에러 처리 여부를 목록화.
4. **첫 실행·업그레이드**: “첫 설치”와 “기존 사용자 마이그레이션” 시나리오를 각각 1줄 이상 명세에 적음.

### 3.2 QA

1. **상태 불일치 테스트**: 마이그레이션 직후 `storage` 값과 메모리 스토어/표시 값이 일치하는지 단언.
2. **테스트 격리**: 전역·모듈 스코프 플래그는 `beforeEach` 또는 전용 `reset*ForTests()`와 조합.
3. **async 타이밍**: 스토어 준비 전/후 UI 스냅샷 또는 행동 차이가 있는지 최소 1케이스.

### 3.3 보안·제품 (선택)

1. 사용자 파일·붙여넣기 입력에 **크기·형식 제한**이 권고에 포함되면 수용 기준 또는 후속 태스크로 명시.
