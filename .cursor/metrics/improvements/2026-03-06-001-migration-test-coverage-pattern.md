## 4. 개선 제안

### 4.1 developer.md - 코드 이관/마이그레이션 시 테스트 규칙 추가

**추가 위치**: "리팩토링" 섹션 근처 또는 "라이브러리 마이그레이션 시" 섹션과 구분하여 별도 섹션

**제안 내용**:

```markdown
### 코드 이관/마이그레이션 시 (필수)

옛날 버전의 코드를 새 버전으로 이관할 때:

1. **옛날 소스의 분기(branch) 목록 파악**
   - 옛날 코드의 if/else, switch, try/catch 등 모든 분기 경로 식별
   - 옛날 코드의 엣지 케이스(빈 배열, null, undefined 등) 처리 확인

2. **테스트 보강 계획 수립**
   - 이관된 코드가 옛날 버전의 **모든 분기**를 커버하는 테스트가 있는지 확인
   - 부재 시: 테스트 작성 또는 QA 에이전트에게 "옛날 분기 기준 테스트 보강" 요청

3. **새 스토어/서비스 모듈 생성 시**
   - 생성 시점에 **단위 테스트 필수** 작성
   - Svelte 스토어($state, store 패턴): subscribe, get, set, reset 등 모든 동작 테스트
   - 서비스 모듈: 공개 함수별 단위 테스트
```

### 4.2 qa.md - 마이그레이션 태스크 시 검증 규칙 추가

**추가 위치**: "테스트 전략 수립" 또는 "코드 리뷰 체크리스트" 섹션

**제안 내용**:

```markdown
### 마이그레이션/코드 이관 태스크 시 (필수)

옛날 버전에서 새 버전으로 코드가 이관된 경우:

1. **옛날 소스의 분기 커버리지 검증**
   - 옛날 코드의 모든 분기(if/else, switch, 경계값)가 테스트로 커버되는지 확인
   - 옛날 코드에 있던 엣지 케이스 처리가 테스트에 반영되었는지 확인

2. **새 스토어/모듈 단위 테스트 체크리스트**
   - 새로 생성된 스토어: subscribe, get, set, reset 등 모든 동작에 대한 테스트 존재 확인
   - 새로 생성된 서비스 모듈: 공개 함수별 단위 테스트 존재 확인
   - 부재 시: 테스트 작성 또는 Developer 에이전트에게 "옛날 분기 기준 테스트 보강" 요청
```

### 4.3 system-improvement.md - 교훈 기반 에러 패턴 매핑 추가

**추가 위치**: "교훈 기반 에러 패턴 분석" (2단계 병목 분석)

## 6. 검증 방법

마이그레이션/코드 이관 태스크 시:

1. **Developer**: 옛날 소스 파일 경로 확인 → 분기 목록 추출 → 테스트에 반영 여부 체크
2. **QA**: 새 스토어/모듈 생성 여부 확인 → 단위 테스트 존재 여부 체크
3. **QA**: 옛날 버전 기준 분기 커버리지 검증 (이관된 코드와 옛날 코드 비교)
