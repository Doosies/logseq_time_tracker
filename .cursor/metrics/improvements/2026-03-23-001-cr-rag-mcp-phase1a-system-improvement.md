## 1. 대규모 Feature(9 마일스톤) 워크플로우 병목

### 관측 (보고서 + 타 사이클 대비)

| 지표 | Phase 1a (보고서) | 단일 패스 Feature 예 (2026-03-17-001) |
|------|-------------------|----------------------------------------|
| developer 호출 | 9 (+ 보안 수정) | 1 |
| qa 호출 | 1 (집약) | 1 |
| security | 1 → 조건부 → 수정 | 1 (통과) |
| git-workflow | 2 (MVP + fix) | 1 |

### 순차 vs 병렬

- **반드시 순차**: 파이프라인 단계(수집 → 요약 → 검증 → 임베딩 → 저장)는 **데이터 의존**이므로 동일 스트림 내 병렬화 이득이 작다.
- **병렬 후보 (제한적)**:
  - **독립 모듈 동시 구현**: 예) `types/` 일괄 + `validators/` 일괄 — 단, 통합 시점에 타입/경계 충돌 비용 발생 가능.
  - **문서화·테스트 스텁**: Feature 워크플로우상 **QA+Docs 병렬**은 이미 main-orchestrator에 존재하나, **마일스톤별로는** 보통 마지막에 QA 1회로 몰리면 병렬 이득이 줄어든다.
- **서브에이전트 호출 최적화**:
  - 마일스톤마다 developer를 부르는 대신, **한 developer 호출에 “다음 마일스톤 범위 + 완료 조건”**을 크게 잡는 방식은 토큰·컨텍스트 한계와 충돌할 수 있음.
  - **타협안**: 3~4개 마일스톤씩 **배치 developer 호출** + 중간에 **경량 검증**(메인의 ReadLints + type-check만)으로 실패 조기 발견.

### 메트릭 관점 병목

- `2026-03-23-001.json`이 실제 호출을 반영하지 않아 **자동 병목 탐지 불가**. 개선 우선순위는 **메트릭 정확도** 자체.

## 2. 서브에이전트 핸드오프 (developer → qa → security → git-workflow)

### 개선 여지

1. **QA 단일 호출**: 9회 developer 후 QA 1회는 **결함 발견 시 수정 범위가 커질** 수 있다. **배치 경계**(예: storage 계층 완료 시점)마다 최소 단위 테스트 또는 qa **스팟 체크**를 고려.
2. **Security 1회 + 사후 High**: 체인은 표준에 맞으나, **“조건부 통과 → developer 재호출 → git-workflow 2회”**로 이어져 **git-workflow 비용**이 증가. 설계·구현 초기에 **경로·외부 입력** 체크리스트를 developer 셀프리뷰에 넣으면 2번째 커밋 확률 감소.
3. **git-workflow**: 과거 사이클(2026-03-17-001)에서 “명령만 준비” 이슈가 있었음. Phase 1a는 2회 커밋으로 **의도적 분리**(feat + fix)로 볼 수 있으나, **한 PR/MR에 fixup** 가능 여부는 팀 정책과 맞추는 것이 좋다.

## 3. 구현 후 보안 발견 패턴

### 원인 분해

- **경로 탈출**: 사용자/설정에서 온 `project_id`를 파일 시스템 경로에 직접 결합 — **고전적 패턴**이나 LLM 구현에서 빈번히 누락.
- **설계 단 Security 조기 호출 효과**:
  - **효과 있음**: “식별자 → 경로 매핑”을 설계 산출물에 명시하면 구현 단계에서 **먼저** `path.resolve` + base jail 검증이 들어갈 가능성이 높아짐.
  - **한계**: ChromaDB HTTP 전용·simple-git 타입 등은 **Security가 아닌 Planner/Developer 도메인** 이슈에 가깝다.

### Windows 특화

- `\`, `..`, 대소문자·드라이브 문자 등은 **OS별로 테스트 케이스**가 가장 확실. 규칙 추가는 **developer 구현 전 체크 + security 스캔 질문** 양쪽에 두는 것이 합리적.
